`ligate-node` is the single binary that runs the chain. Every node syncs
the full state from Celestia DA; whether it produces batches is decided
at runtime by the Postgres lock state (when DbElected mode is active) or
by being the registered preferred sequencer (today's single-sequencer
default). There is no separate "follower" flag. It was removed in
chain v0.2.9 (chain#446); the runtime role check handles the same
"don't propagate submissions" guarantee.

| Deploy type             | Posts batches?                | DA layer                | Typical hardware |
| ----------------------- | ----------------------------- | ----------------------- | ---------------- |
| **Registered sequencer** | Yes (the preferred sequencer) | Celestia Mocha (devnet) | `e2-standard-2` |
| **Read-only sync node** | No (SDK returns 503 on `POST /v1/sequencer/txs` from non-leaders) | Same DA, reads | `e2-medium` |

The public devnet runs a single curated sequencer (Ligate Labs).
Anyone can run a read-only sync node, no permission needed. A
permissionless sequencer set ships at v1+ (see
[ligate-chain#82](https://github.com/ligate-io/ligate-chain/issues/82)).

## Hardware

Reference shape from the
[`public-devnet-deploy.md`](https://github.com/ligate-io/ligate-chain/blob/main/docs/development/public-devnet-deploy.md)
runbook:

| Deploy type        | GCP shape       | vCPU | RAM  | Boot disk | Data disk  | $/mo (sustained) |
| ------------------ | --------------- | ---- | ---- | --------- | ---------- | ---------------- |
| Registered sequencer | `e2-standard-2` | 2  | 8 GB | 20 GB SSD | 100 GB SSD | ~80 USD          |
| Read-only sync node  | `e2-medium`     | 2  | 4 GB | 20 GB SSD | 100 GB SSD | ~30 USD          |

Sequencer headroom covers `ligate-node` + co-located Celestia light
node. Read-only nodes can restore from a snapshot or replay from DA if
state is lost. Ligate Labs runs the public devnet on a single
`e2-standard-2` today (single-sequencer mode, no multi-VM cluster);
the sizing above reflects what a third-party operator would
provision.

<Note>
  OS: **Ubuntu 24.04 LTS**. The released `ligate-node` binary links
  against GLIBC 2.39; Debian 12 ships GLIBC 2.36 and won't load the
  binary without rebuilding from source.
</Note>

Persistent data disk supports **live resize** via `gcloud compute
disks resize` + `resize2fs /dev/sdb`; ext4 grows online without
downtime.

## Install

### Pre-built binary (recommended)

Each tagged release attaches a Linux x86_64 binary. From a fresh
Ubuntu VM:

```bash
# Release tags use the `v` prefix (e.g. v0.2.11); artifact filenames
# omit it (e.g. ligate-node-0.2.11-linux-amd64.tar.gz). Tracked as two
# variables so the curl URL stays correct without shell-substring
# tricks.
LIGATE_TAG=v0.2.11
LIGATE_VERSION=0.2.11
curl -L -o ligate-node.tar.gz \
  https://github.com/ligate-io/ligate-chain/releases/download/$LIGATE_TAG/ligate-node-$LIGATE_VERSION-linux-amd64.tar.gz
tar -xzf ligate-node.tar.gz
sudo mv ligate-node-$LIGATE_VERSION-linux-amd64/ligate-node /usr/local/bin/
ligate-node --version
```

### Container image

Each tagged release publishes a multi-arch (linux/amd64 +
linux/arm64) image to GHCR. Fastest path on any host with Docker:

```bash
docker pull ghcr.io/ligate-io/ligate-chain:0.2.11   # or :latest

# Run against the in-process MockDa (zero external services), config
# + genesis mounted from a host checkout of the chain repo:
docker run --rm -it \
  -v /path/to/ligate-chain/devnet:/var/lib/ligate/devnet:ro \
  -v ligate-state:/var/lib/ligate/state \
  -p 12346:12346 \
  ghcr.io/ligate-io/ligate-chain:0.2.11 \
  --rollup-config-path /var/lib/ligate/devnet/rollup.toml \
  --genesis-config-dir /var/lib/ligate/devnet/genesis
```

The image is debian-slim, runs as the unprivileged `ligate` user
(uid 1000) by default, and exposes `12346` (REST) and `9100`
(Prometheus `/metrics`).

<Note>
  Tag the version explicitly (`:0.2.11`) for reproducible deploys;
  `:latest` follows the most recent release and is fine for casual
  smoke testing but never use it on production VMs.
</Note>

### From source

When you want to track `main` or your platform has no pre-built
binary:

```bash
sudo apt install -y libclang-dev clang  # Ubuntu/Debian
# macOS first install: xcode-select --install

git clone https://github.com/ligate-io/ligate-chain
cd ligate-chain
cargo build --release --bin ligate-node
# Built binary at: target/release/ligate-node
```

Expect 10-15 min cold-cache, ~5 min warm. Pulls ~3 GiB of Rust git
deps.

## Configure

The public devnet config lives at
[`devnet-1/`](https://github.com/ligate-io/ligate-chain/tree/main/devnet-1)
in the chain repo. Two files matter:

- **`devnet-1/celestia.toml`**, rollup config: chain identity, DA
  layer, storage paths, sequencer endpoint
- **`devnet-1/genesis/`**, one JSON per runtime module (accounts,
  attestation, attester_incentives, bank, chain_state,
  operator_incentives, prover_incentives, sequencer_registry)

Clone the chain repo to get them:

```bash
git clone https://github.com/ligate-io/ligate-chain
cd ligate-chain
```

Production deploys override secrets via env vars; never commit them.

## Boot

```bash
SOV_CELESTIA_RPC_URL=ws://127.0.0.1:26658 \
SOV_CELESTIA_GRPC_URL=http://127.0.0.1:9090 \
SOV_CELESTIA_SIGNER_KEY=$(pass celestia/devnet-1-signer) \
ligate-node \
    --da-layer celestia \
    --rollup-config-path devnet-1/celestia.toml \
    --genesis-config-dir devnet-1/genesis
```

This assumes a co-located Celestia light node at the default RPC +
gRPC ports. The light-node setup is covered in the
[Celestia light node runbook](https://github.com/ligate-io/ligate-chain/blob/main/docs/development/runbooks/celestia-light-node.md).

For local development against `MockDa` (no Celestia required), boot
from the simpler `devnet/` config:

```bash
ligate-node  # picks up devnet/rollup.toml + devnet/genesis by default
```

REST binds to `127.0.0.1:12346` for local dev,
`https://rpc.ligate.io` in production.

## Verify

```bash
# Liveness (always 200 if the binary is alive)
curl http://127.0.0.1:12346/health

# Readiness, 200 once synced, 503 while syncing
curl http://127.0.0.1:12346/ready

# Chain identity
curl http://127.0.0.1:12346/v1/rollup/info
# {"chain_id":"ligate-devnet-1","chain_hash":"lsch1...","version":"0.2.11"}
```

## Bootstrap from snapshot

Replaying every Celestia block from genesis takes hours on first
sync. Faster: import a public daily snapshot.

```bash
sudo /opt/ligate/scripts/import-snapshot.sh
```

The script fetches `latest-daily.json` for the chain, downloads the
RocksDB tarball + manifest, verifies SHA-256, stops `ligate-node`,
quarantines existing state, and restores. Full procedure:
[`follower-bootstrap.md`](https://github.com/ligate-io/ligate-chain/blob/main/docs/development/follower-bootstrap.md)
(filename predates the v0.2.9 role-flag removal; the snapshot procedure
itself is unchanged and applies to any node, sequencer or read-only).

## Monitor

- **`/metrics`** on `:9100` (separate port), Prometheus surface. Key
  series: `ligate_state_db_size_bytes`, `ligate_da_blocks_processed_total`,
  `ligate_unconfirmed_txs`, plus standard process / runtime metrics.
- **`/health`** + **`/ready`** on the REST port, orchestrator probes
  (Kubernetes liveness + readiness).
- **Logs**, JSON-structured, level-tunable via `RUST_LOG`.

## What you get for running a node

We get asked this a lot, so the answer up front: **running a sync
node does not earn LGT.** That matches every major rollup operating
today (Optimism, Arbitrum, zkSync, Polygon zkEVM, Base, Starknet
all the same — sync nodes are unpaid). What you get instead:

- **Local verifiability.** Every block, every attestation, every
  schema is re-derived from the Celestia DA blobs by your node, not
  taken on faith from `api.ligate.io`. If our hosted endpoints lie,
  your node sees the discrepancy.
- **Trust-minimised reads.** Your apps, dashboards, or compliance
  tooling can hit `localhost:12346` instead of routing through our
  gateway. Faster, private, no dependency on our uptime.
- **Zero ongoing cost.** Celestia Mocha (the DA layer) is testnet,
  so DA reads are free. The `SOV_CELESTIA_SIGNER_KEY` you configured
  above is just a satisfy-the-config placeholder — your node never
  pays a fee because it never holds the preferred-sequencer lock.
- **Independence from the gateway.** If `rpc.ligate.io` is down for
  maintenance or geographic-routing oddities, your local node
  doesn't care. The chain keeps producing blocks; your node keeps
  syncing.

### Why no LGT for sync nodes

A reward for "run a node that mirrors state" would just be a sybil
printer: anyone can spin up dozens of nodes cheaply and earn for each
one, with no economic work being done. The chains that DO pay node
operators (Cosmos validators, Ethereum stakers, Solana validators)
are paying for *consensus work* with skin in the game — signing
blocks, slashing risk for misbehaviour, real bond requirements. Sync
nodes do none of that.

The protocol's incentive surface is reserved for actors who do real,
verifiable work and post a bond.

## How to earn LGT (today + roadmap)

### Today: attestor sets

This is the one earn-path live on `devnet-1` and open to anyone:

1. Register an attestor set (`ligate register-attestor-set`). Costs a
   one-time bond.
2. Get a schema bound to your set — either register your own schema
   that binds to it, or convince a third-party schema author to
   include your set as a valid attestor surface.
3. Whenever an attestation lands under that schema, the schema's
   `fee_routing_bps` sends a configurable percentage of the
   attestation fee to a configurable address. Set that address to a
   wallet your attestor-set members control, and you earn LGT every
   time the schema is used.

End-to-end:

```bash
# 1. Generate an attestor keypair (or many)
ligate keygen attestor > my-attestor.key

# 2. Register a 1-of-1 attestor set with bond
ligate register-attestor-set --members my-attestor.pub --threshold 1 --bond 1000000000

# 3. Register a schema that routes 5% of attestation fees to your wallet
ligate register-schema \
  --schema-json my-schema.json \
  --attestor-set las1... \
  --fee-routing-bps 500 \
  --fee-routing-addr lig1...

# 4. Submit attestations, or wait for others to submit under your schema
```

`devnet-1` already has live attestor sets earning fees today.
[`docs.ligate.io/cli#ligate-register-attestor-set`](/cli#ligate-register-attestor-set)
walks the flow in detail.

### Roadmap: how node providers will earn

Three mechanisms are in the protocol design and incrementally
shipping. None of them are "run a sync node, get tokens" — all
require posting a bond and doing verifiable work.

1. **Decentralised sequencer set (v1+).** Today devnet-1 runs a
   single registered sequencer (Ligate Labs). The chain code already
   supports a Postgres-coordinated multi-sequencer cluster (see the
   "Multi-sequencer (DbElected)" section below), but the registered
   address set is gated at genesis. In v1 the `sequencer_registry`
   becomes openable: prospective sequencers post a bond
   (`sequencer_bond` in `genesis/sequencer_registry.json`), get
   slashed for double-signing or signing invalid batches, earn from
   the per-batch protocol fee that today goes entirely to the
   Ligate Labs sequencer. Tracking: `ligate-chain` issue #82.
2. **Prover rewards (v1+).** `prover_incentives.json` already exists
   in genesis with a defined fee curve; today the only prover is
   Ligate Labs. v1+ opens prover registration to anyone willing to
   bond and submit valid Risc0 / SP1 proofs of the rollup state
   transition. Provers earn a per-proof fee from the protocol
   treasury. Tracking: `ligate-chain` issue #410 (alongside the
   native DA migration).
3. **Operator incentives for SLA-bound RPC providers (later).** An
   open question for the protocol design: do we want to reward
   external operators who run public RPC mirrors with measurable
   uptime SLAs (Pocket Network style)? Pros: ecosystem diversity.
   Cons: SLA verification is hard, and centralised RPCs (Alchemy
   etc) have shown the market doesn't always reward this. Filed as
   a discussion in `ligate-chain` issue #234 ("public Grafana +
   public RPC providers"). Likely a v1.x feature, not v1.0.

The honest framing for design partners + node operators: the
protocol pays you to do **work** (attest, sequence, prove). It does
not pay you to **exist** (mirror state). Aligns with how every
major rollup has converged.

## Multi-sequencer (DbElected, self-hosted operators)

A Ligate Labs operator can run a federated cluster of `ligate-node`
instances against a shared Postgres-backed leader election. Exactly
one node acts as `BatchProducer` at a time; the others stand by as
`PgSyncReplica` and take over via in-process role transition if the
leader fails. On `devnet-1` we run a three-node cluster and a forced-
failover drill measured **~1.2 seconds** from kill to new leader
producing batches, with no operator intervention.

To activate `DbElected` on your own cluster:

1. Provision a Postgres instance reachable from every sequencer VM
   (we use Cloud SQL HA on devnet-1; any Postgres 14+ works).
2. Add a `[sequencer.preferred.postgres_config]` block to each VM's
   rollup config:
   ```toml
   [sequencer.preferred.postgres_config]
   postgres_connection_string = "postgresql://USER:PASS@HOST:5432/DB?sslmode=require"
   node_id = "ligate-devnet-1-sequencer-N"
   node_role = "DbElected"
   ```
3. Set `bind_host = "0.0.0.0"` in `[runner.http_config]` so the
   gateway VM's Caddy can fan out to the cluster over the VPC.
4. Start `ligate-node` on each VM. The first to win the lock becomes
   `BatchProducer`; the others register in the `nodes` table.

Operator detail lives in the chain repo:

- [`docs/development/runbooks/multi-sequencer-failover.md`](https://github.com/ligate-io/ligate-chain/blob/main/docs/development/runbooks/multi-sequencer-failover.md):
  cluster state read, planned leader replacement, unplanned failure,
  forced takeover, Postgres outage handling.
- [`ops/caddy/Caddyfile`](https://github.com/ligate-io/ligate-chain/blob/main/ops/caddy/Caddyfile):
  leader-aware write routing. `lb_policy first` with an active
  `/v1/sequencer/role` health probe that accepts only response body
  `BatchProducer`, so writes always reach the current leader; reads
  round-robin across synced replicas.

## Known limitations (devnet)

- **No automated DA bridging.** The Celestia signer key is operator-
  managed.
- **Curated validator set.** Permissionless validator entry with
  economic stake lands at v1 alongside the
  [`disputes`](https://github.com/ligate-io/ligate-chain/issues/51)
  module.
- **Read-only-node submissions.** Only the registered preferred
  sequencer (or, under `DbElected`, the current lock-holder) posts
  batches. A sync node's `POST /v1/sequencer/txs` returns 503
  (`PgSyncReplica` role); even if a stale client posted, the blob
  would be filtered at the STF (DA address not in the registry).
  Point clients at `https://rpc.ligate.io` for submissions.
